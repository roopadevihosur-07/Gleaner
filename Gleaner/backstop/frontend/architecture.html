System Architecture & Agent Flow

How Gleaner orchestrates supply chain disruptions using AI agents and specialized tools, coordinating procurement and distribution in one unified loop.

1

🔔 Issue Detection

A supply disruption is detected through multiple channels: real-time monitoring, human alerts, or GPS tracking systems flagging delays.

Live Issues Watcher (Backend)

Monitors supply chain signals: load cuts, delayed shipments, allocation changes, vehicle breakdowns. Triggers the orchestrator.

Input: Real-time signals from maps, tracking systems, supply alerts
Output: Disruption event → triggers Gleaner orchestrator
Role: Listens 24/7, no analysis (just detection)
2

🧠 Orchestrator Agent (Claude AI)

Claude AI wakes up and takes control. It doesn't do math itself—it decides what questions to ask and what tools to call in the right sequence.

Claude Orchestrator

The decision-maker. Reads the disruption, thinks about the problem, and decides which tools to call in what order.

Reasoning: "A TEFAP load got cut. I need to: (1) understand the gap, (2) find vendors, (3) replan routes."
Strategy: Size gap first → then find fill → then replan
Output: Tool calls in sequence, narration of reasoning
3

▤ Gap Analysis Tool

Answers: "How much food is missing? Which agencies lose access? How many meals are at risk?"

Tool: size_gap()

Pure deterministic math. Reads commitment data, inventory, agency allocations. Outputs auditable numbers.

Input: Commitment ID (which load was cut)
Logic: Inventory check → On-hand stock + scheduled inbound − demand − other commitments = gap
Output: {shortfall_lbs: 11300, affected_agencies: 7, meals_at_risk: 9417}
Key Point: NOT hallucinated. Based on real data in `inventory.json` and `commitments.json`
4

$ Procurement Agent

Answers: "Where do we buy replacement food? What's the cheapest reliable option? Can they deliver in time?"

Tool: find_fill()

Searches vendor catalog, checks lead times, calculates cost premium vs. normal procurement.

Input: {category: "protein", shortfall_lbs: 11300, time_window: 4 hours}
Logic: For each vendor:
  • Check if they stock the category
  • Verify lead time < available time
  • Rank by price + reliability score
Output: {vendor: "Sysco", price_per_lb: 200, lead_time: 2hr, premium_avoided: $9800}
Key Point: Avoids panic pricing. Finds reliable option at market rate.
5

⟿ Routing Optimizer

Answers: "How do we deliver the replacement food + adjust existing routes to prevent spoilage?"

Tool: replan_routes()

Takes current route, inserts new vendor pickup, re-sequences stops for access windows and perishability.

Input: {new_load: vendor_location, agencies, vehicle_fleet, perishability}
Logic:   • Add vendor to start of route
  • Re-sequence agencies by window closure time + perishability
  • Calculate distance, fuel cost, estimated spoilage
  • Compare to baseline
Output: {optimized_route: [...], miles: 79, fuel_cost: $45, spoiled_lbs: 0, baseline_miles: 116}
Key Point: Fewer stops = faster delivery = less spoilage. Quantifies the win.
6

🍜 Impact Summarizer

Answers: "In the one metric everyone understands—meals—what did we accomplish?"

Tool: summarize_impact()

Aggregates all the numbers into one clear statement: how many meals are protected?

Input: Gap size, procurement plan, route optimization
Logic:
  • Gap covered: shortfall_lbs ÷ 1.2 = meals
  • Spoilage saved: spoiled_lbs_baseline − spoiled_lbs_optimized ÷ 1.2 = meals
  • Dollars saved (reported separately): (premium avoided + fuel savings)
  • Meals fundable: dollars_saved ÷ food_bank_cost_per_meal
Output: {total_meals_protected: 11000, breakdown: {gap_covered: 9417, spoilage_avoided: 1900}, cost_saved: $9800}
Key Point: Dollars are separate from headline. Never double-count.
7

✓ Human Approval Gate

The plan is presented. A human planner reviews it and decides: Approve or Reject.

Human Decision-Maker

A warehouse lead, operations manager, or planner sees the AI recommendation and makes the final call.

They see: Plain-language recommendation + map of routes + impact card (meals protected)
They decide: "Approve this plan" or "Reject and replan"
Nothing auto-executes. Gleaner augments the human, doesn't replace them.
Key Point: Speed (30 sec) + human wisdom = best outcome
8

🚚 Dispatch & Execution

The approved plan is sent to drivers, procurement contacts vendors, deliveries begin.

Execution (Human Operations)

SMS to drivers, email to vendors, route updates to fleet management. The system's recommendation becomes action.

Outcomes tracked: Delivery confirmations, actual spoilage, actual cost, actual meals delivered
Future learning: Data feeds back for vendor reliability updates, route accuracy, cost modeling

System Architecture Layers

📡 Data Layer (What the system knows)
Commitments
TEFAP allocations, purchase orders, donor loads, lead times
Inventory
Current on-hand by category, expiration dates, storage capacity
Agencies
Partner locations, lat/lng, access windows, equity priority
Vendors
Catalog, pricing, lead time, reliability scores
Fleet
Truck capacity, refrigeration, fuel cost, depot location
🔧 Tool Layer (What the system can do)
size_gap()
Calculate shortfall in lbs → meals at risk
find_fill()
Search vendors → best price + lead time + reliability
replan_routes()
Insert pickup → optimize stops → calculate savings
summarize_impact()
Aggregate into one metric: meals protected
🧠 Orchestration Layer (Claude AI)
Understand
Parse disruption, assess severity, identify scope
Reason
Decide which tools to call and in what order
Execute
Call tools sequentially, wait for results, chain reasoning
Narrate
Translate results into human-readable recommendation
👥 Human Layer (Always in control)
Monitor
Watch live issues feed, see active disruptions
Review
See AI plan on map, read metrics, understand tradeoffs
Decide
Approve plan, request replanning, or override with domain expertise
Execute
Dispatch drivers, contact vendors, manage on-the-ground logistics

🔄 Data Flow Through the System

How information moves from signal to decision:

The Flow: A disruption signal arrives → Claude AI reads and understands it → Claude calls specialized tools (size_gap, find_fill, replan_routes) → Results are aggregated into one metric (meals protected) → A human decision-maker reviews and approves → The approved plan is executed (drivers dispatched, vendors contacted)

Tool Specifications

▤ size_gap() — Gap Analysis

Calculates the impact of a cut/delayed load. How many meals are at risk? Which agencies? What's the shortfall?

commitment_id: "TEFAP-1042" (the cut load)
{
  shortfall_lbs: 11300,
  meals_at_risk: 9417,
  affected_agencies: ["Food Bank 1", "Food Bank 2", ...],
  by_category: { protein: 11300 }
}

Algorithm: Reads inventory.json (on-hand), commitments.json (inbound schedule), calculates net availability. Pure deterministic math, fully auditable.

$ find_fill() — Procurement Search

Finds vendors who can deliver replacement food in time. Ranks by price and reliability. Avoids rush premiums.

category: "protein",
shortfall_lbs: 11300,
time_window: 4 (hours available for delivery)
{
  vendor: "Sysco Purchase Order",
  price_per_lb: 200,
  lead_time_hours: 2,
  reliability_score: 0.95,
  total_cost: $2260000,
  premium_avoided: $9800 (vs. panic buy)
}

Algorithm: Searches vendors.json, filters by (lead_time < window), ranks by (price × reliability). Compares to baseline panic pricing.

⟿ replan_routes() — Routing Optimizer

Takes the current delivery plan and the new vendor pickup. Re-sequences stops to minimize distance, prevent spoilage, honor access windows.

new_vendor_location: {lat, lng},
existing_agencies: [{lat, lng, window_close_time, perishability}, ...],
vehicle_capacity: 50000 lbs
{
  optimized_route: [vendor, agency_2, agency_5, agency_1, ...],
  optimized_miles: 79,
  optimized_spoiled_lbs: 0,
  baseline_miles: 116,
  baseline_spoiled_lbs: 1900,
  miles_saved: 37,
  fuel_cost_saved: $45
}

Algorithm: Greedy nearest-neighbor with constraints (time windows, perishability). Calculates distance via lat/lng, spoilage via time-to-arrival + product shelf life.

🍜 summarize_impact() — Impact Aggregation

Combines all results into one metric: meals protected. Converts dollars to additional meals the food bank can fund.

gap_meals: 9417,
spoilage_saved_meals: 1900,
cost_saved: $9800,
food_bank_cost_per_meal: $1.78
{
  total_meals_protected: 11317,
  breakdown: {
    gap_covered: 9417,
    spoilage_avoided: 1900
  },
  cost_saved: $9800,
  meals_fundable_with_savings: 5506
}

Design Choice: Dollars are reported SEPARATELY and converted via the food bank's own cost/meal. Never inflated into headline.

Why This Architecture Works

🤖 → 🔧

AI Orchestrates, Tools Calculate

Claude (reasoning) decides what to ask. Tools (math) answer auditably. No hallucination, no guessing. Numbers are real.

⚡

Speed Through Parallelization

Some tools could run in parallel (gap analysis and vendor search). Sequential here for clarity. Easy to optimize.

👥

Human Always Decides

AI proposes. Humans approve. No auto-execution. Gleaner is a tool for planners, not a replacement. Augment, don't replace.

🍜

One Metric: Meals

Not miles, not dollars, not logistics points. Real food to real people. Procurement team, distribution team, food bank CEO—everyone speaks meals.

📊

Data-Driven, Not Assumption-Based

Every number comes from data files (agencies.json, vendors.json, inventory.json). Swap in real data, get real results. No magic.

🔄

Deterministic Fallback

Same logic runs with or without Claude. Offline demo mode works. Reliability first, AI second. Hybrid approach.

Gleaner — System Architecture

Each agent has one job. Together, they coordinate disruption-to-dispatch in 30 seconds. Human approves. Everything is auditable.
